PRD: Professional Portfolio Website (GitHub Pages) for Senior Implementation Manager

Document Owner: Jon (Portfolio owner)
Product Type: Personal professional portfolio website
Platform: GitHub Pages (static site)
Status: Draft PRD (MVP + launch guidance)
Target Launch: TBD (see timeline section)

1) Context

This product is a professional portfolio website for a U.S.-based Senior Implementation Manager candidate in healthcare technology. The site is intended to support career advancement into enterprise-facing implementation leadership roles (with Cohere Health’s Senior Implementation Manager role as the reference persona and tone benchmark).

The site should present a clear, credible professional narrative that demonstrates:

end-to-end implementation leadership,

client-facing communication,

cross-functional execution,

operational rigor,

healthcare domain fluency,

and the ability to translate complex work into actionable plans.

The structure should be inspired by the simplicity and scan-ability of the referenced GitHub Pages site (clear navigation, straightforward content sections, project-style entries, contact links), while adapting the content and tone to implementation/program delivery rather than technical research projects. The reference site uses a simple top navigation, project listing, and contact links that are easy to scan and maintain.

Cohere Health’s current Senior Implementation Manager posting emphasizes end-to-end implementation ownership, pre-sales/solutioning partnership, cross-functional coordination, client launch success, risk tracking, project plan management, and building repeatable implementation playbooks—these are the persona anchors this portfolio should reflect.

2) Problem Statement
Problem

A resume and LinkedIn profile alone often do not fully demonstrate how an implementation leader:

structures complex programs,

manages risks and dependencies,

communicates across technical/commercial/clinical stakeholders,

and drives repeatable outcomes.

For senior healthcare implementation roles, hiring teams need evidence of leadership style, systems thinking, execution discipline, and communication clarity, not just job titles.

Why this matters

Without a purpose-built portfolio:

the candidate’s implementation strengths may appear generic,

nuanced project leadership impact may be under-communicated,

and opportunities to demonstrate strategic thinking and operational maturity may be lost.

3) Vision Statement

Create a professional portfolio website that makes the candidate feel like a credible, enterprise-ready Senior Implementation Manager in healthcare technology within the first 60–90 seconds of review.

The site should communicate:

Trustworthiness (organized, polished, accurate),

Execution capability (plans, outcomes, risk management, cross-functional leadership),

Healthcare implementation relevance (payer/provider workflows, launches, operational rigor),

Communication excellence (clear, concise, structured narrative),

Technical fluency (enough to partner with product/engineering without over-positioning as a developer).

9) Non-Goals (Explicit)

To keep MVP focused, the following are non-goals:

Building a complex web app

This is a professional portfolio, not a SaaS product.

Demonstrating full-stack engineering depth

Technical fluency should be visible, but the primary signal is implementation leadership.

Publishing sensitive client details

No confidential data, protected information, or proprietary implementation specifics.

Over-optimizing visual effects

Clean, readable, credible presentation is more important than animation-heavy design.

Creating a long-form content engine on day one

Thought leadership can begin with a few strong pieces instead of a full editorial cadence.

10) User Experience (UX) Principles
UX Goals

Fast comprehension: visitors understand value proposition quickly.

High scan-ability: content chunked for busy recruiters/hiring managers.

Professional credibility: polished but restrained visuals.

Low friction navigation: minimal clicks to core evidence and contact.

Accessible readability: legible typography, clear hierarchy, mobile-friendly.

UX Principles

Clarity over cleverness

Prefer direct labels like “Experience,” “Projects,” “Contact.”

Evidence over claims

Use outcomes, examples, and structured case summaries instead of broad statements.

Progressive disclosure

Show concise summaries first; provide expandable/detail pages if needed.

Consistency

Reuse the same template format for project/case entries (context → role → actions → outcome).

Executive-grade writing

Short paragraphs, strong headers, bullet points where appropriate.

Representative User Journeys
Journey A: Recruiter (2–3 minute scan)

Lands on homepage → reads positioning summary → scans experience and projects → opens LinkedIn/resume → contacts candidate.

Journey B: Hiring Manager (5–10 minute evaluation)

Lands on homepage → reads role fit → reviews 2–3 implementation case studies → assesses thought leadership tone → views contact/resume → schedules interview.

Journey C: Cross-functional interviewer

Reviews project/case entries to understand how candidate handles product/engineering/clinical collaboration and risk management.

13) Technical Requirements
13.1 Hosting & Deployment (GitHub Pages)

The site will be hosted on GitHub Pages for simplicity, low cost, and version-controlled maintenance. GitHub supports publishing from a branch or via GitHub Actions, and recommends branch-based publishing for simple setups and custom Actions workflows when a custom build process is needed.

Recommended MVP deployment approach (Option A: simplest)

Static site in repository root (or /docs)

Publish using GitHub Pages “Deploy from a branch”

No build step required (plain HTML/CSS/JS)

Recommended scalable approach (Option B: preferred if using framework)

GitHub Pages with GitHub Actions custom workflow

Build + deploy from main branch

Keeps source and built artifacts organized

Supports static site generators/frameworks cleanly

GitHub docs explicitly support custom workflows for Pages deployments and provide templates for common static site generators.

13.2 Framework / Stack Options (Recommended)
MVP recommendation

Choose one of these based on desired complexity:

Option 1 (Lowest complexity):

HTML + CSS + minimal JavaScript

Best for speed and long-term simplicity

Option 2 (Balanced maintainability):

Static site generator (e.g., Jekyll or Eleventy)

Good for reusable templates (projects/case studies)

Option 3 (More polish/flexibility):

React-based static export (if comfortable)

Use only if you want component reuse and future expansion

PRD recommendation

For this portfolio’s goals, Option 1 or 2 is sufficient and preferred for MVP:

fast to build,

easy to maintain,

minimal deployment complexity,

excellent fit for GitHub Pages.

16) Risks and Mitigations
Risk 1: Content feels too generic

Mitigation: Use structured case studies with concrete outcomes and repeatable lessons.

Risk 2: Overly dense text reduces recruiter usability

Mitigation: Enforce scan-friendly layout, summaries, and section templates.

Risk 3: Site looks technical but not role-aligned

Mitigation: Keep positioning centered on implementation leadership and client delivery outcomes.

Risk 4: Scope creep delays launch

Mitigation: Ship MVP first (core sections + 3 case studies), defer polish features.

Risk 5: Confidentiality issues in examples

Mitigation: Sanitize names, metrics, and implementation details; use generalized descriptions where necessary.
